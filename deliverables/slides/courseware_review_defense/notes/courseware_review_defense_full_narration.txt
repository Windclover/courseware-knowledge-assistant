大家好，我汇报的项目是“课件智能知识点复习助手”。这是一个面向课程复习和答辩准备的本地 Web 应用。它的核心目标，是把老师给出的 PPT 或 PDF 课件，自动转化成可以复习、可以追问、可以测试、还可以导出的学习工作台。相比传统只靠翻课件和手写笔记的方式，这个项目更强调“从课件到理解，再到练习和检验”的完整复习闭环。

这个项目来自一个很现实的学习场景。第一，课程课件往往内容很多，特别是数学或算法类课程，页面多、信息密，学生很难快速抓住重点。第二，真正需要掌握的内容往往分散在章节讲解、公式、例题和老师口头说明里，复习时需要自己重新整理。第三，传统课件本身缺少互动能力，学生看到不理解的地方，不能马上追问，也很难把上下文连续保留下来。第四，练习题和测试题往往需要老师额外准备，复习效率不高。所以我们希望把静态课件，升级成一个可交互、可测试的学习工具。

从输入输出的角度来看，这个项目的输入非常简单，就是课程的 PPTX 或文本型 PDF。系统接收到课件以后，会自动完成文本解析、章节聚合、学习笔记生成、学习任务规划和测试题生成。最后用户得到的不是一份简单摘要，而是一个完整的学习工作台，包括结构化笔记、Learning Board、研究对话区和导出结果。它的价值在于，帮助学生从“被动看课件”转向“主动理解、主动提问、主动练习”，尤其适合课程复习和答辩准备。

系统整体可以概括为四个能力模块。第一是课件解析，负责把 PPT 和 PDF 中的内容提取出来，并组织成可计算的结构。第二是笔记生成，把原始课件内容整理成章节讲解、关键知识点、公式和例题。第三是对话问答，用户可以围绕课件和学习笔记继续追问，而不是重新回去翻原始材料。第四是测试与导出，用户完成复习任务之后进入测试环境，最后还能把笔记、任务完成情况和答题结果统一导出。

系统工作流从上传课件开始。上传之后，后端先做文本解析和去噪，接着完成章节切分。完成章节组织后，模型会基于这些内容生成学习笔记和 Learning Board。用户随后进入研究对话区，可以针对课件和笔记继续提问。当 Learning Board 中的任务完成后，系统自动进入测试环境，生成选择题、填空题和计算题。最后，用户可以把整个复习过程导出成 Markdown 复习记录。也就是说，这不是一组孤立功能，而是一条完整串联的学习流程。

在界面上，我们采用的是一个三栏式工作台。左侧是资料区，用来放上传入口、课件列表和笔记摘要。中间是研究对话区，也是整个产品的主舞台，因为用户大多数操作都会围绕提问和理解展开。右侧是学习任务台和测试区，负责把系统生成的摘要、概念、练习和复习路径组织起来，并在任务完成之后切换到测试环境。这种布局的目的是让“复习”和“提问”成为主流程，而不是把页面做成很多功能卡片的堆叠。

第一个核心功能是课件解析与章节组织。系统支持两类输入，一类是 PPTX，另一类是文本型 PDF。对于 PDF，难点并不是单纯抽文本，而是很多课件会带有重复页眉、目录导航和页码信息，这些噪声会严重影响章节识别。我们在真实案例“罚函数法.pdf”里就遇到了这个问题：原来整份 24 页课件会被错误地聚成 1 个 section。后来我们改成了“先做按页行文本提取，再识别高频噪声，再从前几行里找真实短标题”的方法，最后把它修到了 11 个 section，说明解析层已经能处理这种带目录噪声的数学课件。

第二个核心功能是学习笔记生成。我们一开始只生成一小段说明文字，效果很差，根本不像学生真正会拿来复习的笔记。后来我们把笔记结构升级成五个部分：章节概述、详细讲解、关键知识点、核心公式，以及例题与计算步骤。特别是对于数学类课件，系统不能只讲概念，还要把公式和例题整理出来。以罚函数法这份课件为例，系统现在能够把外点罚函数和内点罚函数相关公式提取出来，并把等式约束举例、不等式约束举例这样的内容整理成更适合复习的讲解结构。

第三个核心功能是研究对话区。这里不是一个普通的聊天框，而是围绕课件与学习笔记的结构化问答系统。我们专门做了问题类型识别，会区分用户是在问摘要、概念、公式、练习题，还是在承接上一轮对话继续追问。对于“整份课件的重点是什么”这类问题，系统会走全局上下文；对于“用公式讲一下”这类问题，会优先走公式上下文；对于“帮我出个练习题”“怎么解决”这类问题，会把上一轮例题承接下来，而不是每次都退回去复述整份课件。这样做的结果是，对话区开始真正承担“理解”和“继续推导”的角色。

第四个核心功能是学习任务与测试环境。Learning Board 不是简单列目录，而是把系统生成的内容组织成四块：摘要、概念、练习和复习路径。这样用户可以知道应该先看什么、后看什么，以及每一步应该解决什么问题。当任务完成以后，系统进入测试环境，生成选择题、填空题和计算题。这里有一个重要改进：最开始系统只能出概念题，后来我们把测试逻辑扩展到了计算题，让它能够根据课件中的公式和例题生成更贴合课程内容的问题。交卷之后，还会显示正确与否、标准答案、解析和参考步骤。

在技术架构上，前端使用的是 React、TypeScript 和 Vite，后端使用 FastAPI，底层存储用 SQLite。文本解析层使用 python-pptx 和 PyMuPDF，分别处理 PPTX 和 PDF。模型调用层目前走 OpenAI compatible 的 Responses API，默认模型是 gpt-5.4，推理强度是 medium。整个系统通过几个核心接口完成前后端联动，例如上传课件、获取文档详情、研究对话、生成测试以及下载 Markdown 结果。这样的架构保证了项目既能本地运行，又便于后续继续扩展。

在实现过程中，我们重点解决了四个问题。第一个是 PDF 的目录噪声问题，它会直接破坏章节切分，我们通过页面去噪和真实标题识别修复了这一点。第二个是 Markdown 笔记过短、缺少公式和例题，我们通过扩展章节结构，把讲解、公式和例题统一写进笔记中。第三个是测试题最初全是概念题，后来通过 assessment 扩展和计算题 fallback，让数学类课件也能产出计算题。第四个是研究对话区很容易退化成泛化总结，我们后来加入了问题类型识别、全局与局部检索分流，以及练习题承接逻辑，才让对话质量真正提升。

从当前结果来看，这个项目已经不只是原型。首先，在真实数学课件“罚函数法.pdf”上，章节切分从原来的 1 个 section 修复到了 11 个 section。其次，学习笔记长度提升到了 11025 字左右，而且已经包含公式和例题步骤。再次，测试环境里可以稳定生成 2 道计算题，不再全是概念题。最后，研究对话区经过多轮优化以后，已经能更稳定地回答公式问题、承接上一轮练习题并给出继续求解的答案。这些结果说明项目已经具备了实际展示和继续打磨的基础。

最后做一个总结。课件智能知识点复习助手的核心价值，在于把传统静态课件转化成一个完整的学习闭环：先解析，再生成笔记，再组织任务，再进入对话和测试，最后还能导出复习结果。对于后续工作，我认为有三个方向最值得继续做：第一，支持 OCR，让扫描版 PDF 也能处理；第二，进一步加强公式规范化，让数学表达更标准；第三，继续增强数学推导和自动评测能力。我的汇报到这里结束，谢谢大家，欢迎提问。
